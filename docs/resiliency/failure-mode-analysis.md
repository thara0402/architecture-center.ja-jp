# <a name="failure-mode-analysis"></a><span data-ttu-id="27ad1-103">障害モード分析</span><span class="sxs-lookup"><span data-stu-id="27ad1-103">Failure mode analysis</span></span>
[!INCLUDE [header](../_includes/header.md)]

<span data-ttu-id="27ad1-104">障害モード分析 (FMA) は、システムで考えられる障害点を特定することによって、システムに回復力を持たせるためのプロセスです。</span><span class="sxs-lookup"><span data-stu-id="27ad1-104">Failure mode analysis (FMA) is a process for building resiliency into a system, by identifying possible failure points in the system.</span></span> <span data-ttu-id="27ad1-105">FMA をアーキテクチャおよび設計フェーズで行って、最初からシステムに障害復旧を組み込むことができるようにする必要があります。</span><span class="sxs-lookup"><span data-stu-id="27ad1-105">The FMA should be part of the architecture and design phases, so that you can build failure recovery into the system from the beginning.</span></span>

<span data-ttu-id="27ad1-106">FMA の一般的な実施プロセスを次に示します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-106">Here is the general process to conduct an FMA:</span></span>

1. <span data-ttu-id="27ad1-107">システム内のすべてのコンポーネントを明らかにします。</span><span class="sxs-lookup"><span data-stu-id="27ad1-107">Identify all of the components in the system.</span></span> <span data-ttu-id="27ad1-108">ID プロバイダーやサード パーティのサービスなど、外部の依存関係を含みます。</span><span class="sxs-lookup"><span data-stu-id="27ad1-108">Include external dependencies, such as as identity providers, third-party services, and so on.</span></span>   
2. <span data-ttu-id="27ad1-109">コンポーネントごとに発生する可能性のある障害を特定します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-109">For each component, identify potential failures that could occur.</span></span> <span data-ttu-id="27ad1-110">1 つのコンポーネントに複数の障害モードが存在する場合があります。</span><span class="sxs-lookup"><span data-stu-id="27ad1-110">A single component may have more than one failure mode.</span></span> <span data-ttu-id="27ad1-111">たとえば、影響と可能な軽減策が異なるため、読み取り障害と書き込み障害は別のものとして検討する必要があります。</span><span class="sxs-lookup"><span data-stu-id="27ad1-111">For example, you should consider read failures and write failures separately, because the impact and possible mitigations will be different.</span></span>
3. <span data-ttu-id="27ad1-112">総合的なリスクに従って、各障害モードを評価します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-112">Rate each failure mode according to its overall risk.</span></span> <span data-ttu-id="27ad1-113">次の点を考慮します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-113">Consider these factors:</span></span>  

   * <span data-ttu-id="27ad1-114">障害の可能性はどれくらいか。</span><span class="sxs-lookup"><span data-stu-id="27ad1-114">What is the likelihood of the failure.</span></span> <span data-ttu-id="27ad1-115">比較的よくあることか。</span><span class="sxs-lookup"><span data-stu-id="27ad1-115">Is it relatively common?</span></span> <span data-ttu-id="27ad1-116">非常にまれなことか。</span><span class="sxs-lookup"><span data-stu-id="27ad1-116">Extrememly rare?</span></span> <span data-ttu-id="27ad1-117">正確な数値は必要はありません。目的は、優先順位をランク付けすることです。</span><span class="sxs-lookup"><span data-stu-id="27ad1-117">You don't need exact numbers; the purpose is to help rank the priority.</span></span>
   * <span data-ttu-id="27ad1-118">可用性、データの損失、金銭的コスト、および業務の中断の観点から、アプリケーションに対してどの程度の影響がありますか。</span><span class="sxs-lookup"><span data-stu-id="27ad1-118">What is the impact on the application, in terms of availability, data loss, monetary cost, and business disruption?</span></span>
4. <span data-ttu-id="27ad1-119">各障害モードについて、アプリケーションがどのように対応および復旧するかを決定します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-119">For each failure mode, determine how the application will respond and recover.</span></span> <span data-ttu-id="27ad1-120">コストとアプリケーションの複雑さのトレードオフを検討します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-120">Consider tradeoffs in cost and application complexity.</span></span>   

<span data-ttu-id="27ad1-121">この記事では、FMA プロセスの出発点として、起こりうる障害モードの一覧とその軽減方法について説明します。</span><span class="sxs-lookup"><span data-stu-id="27ad1-121">As a starting point for your FMA process, this article contains a catalog of potential failure modes and their mitigations.</span></span> <span data-ttu-id="27ad1-122">カタログは、テクノロジまたは Azure サービス、およびアプリケーション レベルの設計の一般的なカテゴリで分けられています。</span><span class="sxs-lookup"><span data-stu-id="27ad1-122">The catalog is organized by technology or Azure service, plus a general category for application-level design.</span></span> <span data-ttu-id="27ad1-123">カタログは完全ではありませんが、主要な Azure サービスの多くをカバーしています。</span><span class="sxs-lookup"><span data-stu-id="27ad1-123">The catalog is not exhaustive, but covers many of the core Azure services.</span></span>
